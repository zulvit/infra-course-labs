# Лабораторная работа 6

## Тема

Docker Compose

## Цель работы

Изучить создание и запуск многоконтейнерного приложения при помощи Docker Compose, а также проверить взаимодействие frontend, backend и базы данных.

## Ход выполнения

### 1. Подготовка проекта

Был подготовлен compose-проект, включающий три сервиса:

- `front` — веб-сервер `nginx`;
- `back` — backend-приложение на Go;
- `db` — база данных PostgreSQL.

Для выполнения лабораторной были созданы следующие файлы:

- `work/compose.yaml`
- `work/index.html`
- `work/default.conf`
- `work/.env`
- `work/users.sql`
- `work/main.go`
- `work/Dockerfile`
- `work/run.sh`

### 2. Запуск окружения Docker Compose

Проект был собран и запущен через Docker Compose с автоматической сборкой backend-образа.

Результат:

- все сервисы успешно стартовали;
- compose-окружение было развёрнуто без ошибок;
- состояние контейнеров было дополнительно проверено служебными командами Docker Compose.

### 3. Проверка frontend

После запуска был выполнен HTTP-запрос к frontend-сервису.

Результат:

- страница успешно открывалась;
- отображалась подготовленная HTML-страница;
- frontend корректно обращался к backend по маршруту `/back`.

Фрагмент HTML-ответа:

```html
<title>Lab6 Compose</title>
<h2>Users</h2>
```

### 4. Проверка backend и PostgreSQL

Backend-сервис возвращал список пользователей из базы данных PostgreSQL в формате JSON.  
После этого в таблицу была добавлена новая запись через `psql`.

Результат:

- до вставки отображались начальные записи;
- после добавления нового пользователя API возвращал обновлённый список;
- взаимодействие между frontend, backend и базой данных работало корректно.

Ответ API до вставки:

```json
[{"id":1,"name":"john","email":"john@example.com"},{"id":2,"name":"mary","email":"mary@example.com"}]
```

Ответ API после вставки:

```json
[{"id":1,"name":"john","email":"john@example.com"},{"id":2,"name":"mary","email":"mary@example.com"},{"id":3,"name":"alex","email":"alex@mail.ru"}]
```

### 5. Проверка сохранности данных

Для хранения данных PostgreSQL был подключён именованный том `db-data`.  
После этого контейнеры были удалены и заново подняты.

Результат:

- данные в базе сохранились после пересоздания контейнеров;
- добавленная ранее запись осталась доступной;
- работа постоянного тома была подтверждена.

Ответ API после пересоздания контейнеров:

```json
[{"id":1,"name":"john","email":"john@example.com"},{"id":2,"name":"mary","email":"mary@example.com"},{"id":3,"name":"alex","email":"alex@mail.ru"}]
```

### 6. Проверка сетевой изоляции

Контейнеры были разделены на две сети:

- `front`;
- `db`.

Сервис `back` был подключён к обеим сетям.

Результат:

- frontend не имел прямого доступа к контейнеру базы данных;
- попытка обращения из `front` к `db` завершалась ошибкой;
- требуемая изоляция между сервисами была обеспечена.

Фрагмент ошибки:

```text
wget: bad address 'db:5432'
```

## Результаты

В ходе лабораторной работы были успешно проверены:

- запуск многоконтейнерного приложения через Docker Compose;
- взаимодействие frontend, backend и PostgreSQL;
- работа volume для постоянного хранения данных;
- разделение сервисов по сетям.

## Вывод

В результате выполнения лабораторной работы был развёрнут и проверен многоконтейнерный проект на основе Docker Compose.  
Подтверждена корректная работа всех сервисов, сохранность данных в базе после пересоздания контейнеров и сетевая изоляция между frontend и базой данных.
