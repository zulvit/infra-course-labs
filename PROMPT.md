# Промпт для автономного выполнения лабораторных (CLI-агент)

Ты выполняешь DevOps лабораторные работы **полностью автономно**.

## Источник лабораторных

Репозиторий: **https://github.com/zulvit/infra-course-labs**

Если ты уже находишься **внутри** клонированного репозитория, не клонируй повторно — работай из корня проекта. Иначе:

```bash
git clone https://github.com/zulvit/infra-course-labs
cd infra-course-labs
```

Если хост — **Windows**, для шагов методички в стиле Linux (`apt`, `bash`, `docker`, `ansible`) работай в **WSL2** (например Ubuntu): открой `wsl`, клонируй репозиторий внутри дистрибутива и выполняй команды оттуда; **Docker Desktop** с интеграцией WSL удобен для лаб с Docker/Compose; **Vagrant + VirtualBox** надёжнее на физическом ПК или отдельной ВМ с включённой виртуализацией в BIOS (VT-x/AMD-V), избегай «ВМ внутри ВМ» без nested virtualization. В отчётах указывай пути **относительно корня репозитория** (`lab1/vagrant/Vagrantfile`, `lab7/report.md`), не вставляй чужие абсолютные пути вида `/home/username/...`.

## Задача

1. Получить код репозитория (клонировать или использовать текущий каталог).
2. Найти лабораторные: каждая в своей папке `lab1`, `lab2`, …
3. Выполнять их **по одной**, не смешивая контекст разных лаб.

## Текущий статус проекта

- Репозиторий уже подготовлен и содержит материалы по `lab1`–`lab8`.
- `lab1` подготовлена и частично проверена: есть `lab1/vagrant/Vagrantfile` и `lab1/vagrant/main.go`; в среде без nested virtualization зафиксирована блокировка VirtualBox: `AMD-V is not available (VERR_SVM_NO_SVM)`.
- `lab2` задокументирована как заблокированная тем же ограничением: без рабочего VirtualBox/Vagrant нельзя поднять `front`/`back`/`db` и собрать боксы.
- `lab3` в одном из прогонов выполнялась в адаптированном виде через `udocker` (обход без полноценного Docker daemon).
- `lab4` частично через `kaniko` + `udocker`: сборка образа, локальный registry и push; не все runtime-сценарии курса эквивалентны «чистому» Docker.
- `lab5` на одном хосте с Docker: bind mount, named volumes, общий том, пользовательский bridge и `host` network; multi-host NFS / IPVLAN на нескольких ВМ могли не воспроизводиться.
- `lab6` на Docker Compose v2: стек `front`/`back`/`db`, сети и персистентность тома — по отчётам прогонов.
- `lab7` и `lab8` часто остаются с пустыми шаблонами в `lab7/report.md` и `lab8/report.md` — приоритет для следующих запусков агента.

## Что делать дальше на машине, где доступна виртуалка

- Не начинать с нуля: использовать текущее состояние репозитория как базу и продолжать в уже подготовленных каталогах.
- Перепроверить `lab1` на хосте или ВМ с рабочей VirtualBox и виртуализацией в BIOS: `vagrant up`, `vagrant ssh`, `curl localhost:8080`; при успехе обновить `lab1/report.md` (снять статус блокировки, добавить проверки).
- Выполнить `lab2` в формате курса: боксы `front`/`back`/`db`, multi-node стенд, `http://localhost:8888` — отразить в `lab2/report.md`.
- При необходимости строгого совпадения с курсом довести multi-host части `lab5` (NFS, при необходимости `ipvlan` на нескольких ВМ).
- Выполнить `lab7` и `lab8`, заполнить `lab7/report.md` и `lab8/report.md`.
- При полноценном Docker и правах root по желанию выровнять `lab3` и `lab4` под классический сценарий курса (вторично после `lab1`, `lab2`, `lab7`, `lab8`).

## Общие правила

### 0. Подготовка среды

- Клонирование (если репозитория ещё нет):

  ```bash
  git clone https://github.com/zulvit/infra-course-labs
  cd infra-course-labs
  ```

- Перед стартом **конкретной** лабораторной перейди в её папку, например `lab1/`.
- В каталоге лабораторной создай при необходимости структуру для артефактов:

  ```bash
  mkdir -p assets/screenshots artifacts/logs
  ```

### 1. Анализ

- Прочитай `task.md` в текущей папке лабораторной.
- Составь план шагов (кратко, в отчёте или перед выполнением).

### 2. Устаревшие инструкции

- Если команды или зависимости из `task.md` устарели (EOL образа, переименование пакета, смена синтаксиса):
  - замени на **актуальные** эквиваленты;
  - зафиксируй в `report.md`: что было, что стало, почему.

### 3. Execution loop

- Цикл: шаг → команда → результат → анализ → исправление → повтор.
- **Не более 3 попыток** на один и тот же шаг.
- Не повторяй **одинаковые** команды без изменений (аргументов, окружения, контекста).

### 4. Выполнение

- Ориентируйся на **Linux CLI** (`apt`, `docker`, `systemctl`, `vagrant`, и т.д.); на Windows используй **WSL2** или эквивалент (см. абзац после блока `git clone` в начале документа).
- Работай в рамках **папки текущей лабораторной** (`labN/`); не разноси артефакты соседних лаб без необходимости.
- В `report.md` ссылайся на файлы **от корня репозитория** (`labN/...`), без абсолютных путей чужой машины.

### 5. Ошибки

- Разбирай `stderr` и код выхода.
- Пробуй разумные альтернативы.
- Если за **3 попытки** шаг не удался:
  - опиши проблему, команды и вывод в `report.md`;
  - продолжай дальше только если оставшиеся шаги не зависят от проваленного (иначе явно укажи блокировку).

### 6. Валидация

После значимых шагов проверяй по смыслу задания:

- сервис запущен?
- порт слушается / доступен?
- `curl` (или аналог) даёт ожидаемый ответ?
- конфиг применился?

### 7. Скриншоты

- Сохраняй в каталог лабораторной:

  `assets/screenshots/`

- Пример (если доступен `scrot`):

  ```bash
  scrot assets/screenshots/stepNN.png
  ```

- Если графической среды нет или `scrot` недоступен: сохрани **текстовый снимок** (например, вывод `curl -v`, `systemctl status`, `docker ps`) в `artifacts/logs/` и укажи в `report.md`, что скриншот заменён логом.

### 8. Логи

- Сохраняй вывод команд в:

  `artifacts/logs/`

- Для ключевых команд:

  ```bash
  command > artifacts/logs/stepNN.out 2> artifacts/logs/stepNN.err
  ```

- По возможности веди общий журнал сессии (например, `script` или копирование блока вывода в один файл) и путь укажи в отчёте.

### 9. Отчёт

Обновляй `report.md` по ходу работы:

- шаги;
- команды;
- результат и проверки;
- ошибки и исправления;
- пути к скриншотам (или заменяющим логам);
- пути к логам.

### 10. Безопасность

- Не выполняй **разрушающие** команды вне рамок лаборатории (без явной необходимости и понимания последствий): массовое удаление данных, `rm -rf /`, форматирование дисков, сброс чужих кластеров и т.п.
- Минимизируй постоянные изменения **хостовой** системы; по возможности изолируй (ВМ, контейнер).

### 11. Завершение

Лабораторная считается завершённой, если:

- требования из `task.md` выполнены **или** честно задокументированы как невыполнимые в текущей среде;
- проверки пройдены там, где это возможно;
- есть актуальный `report.md`;
- есть скриншоты **или** осознанная замена логами (см. п. 7);
- ключевые команды отражены в логах (п. 8).

3. Начать выполнение **первой** лабораторной по `task.md`.
