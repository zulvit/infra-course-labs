## Лабораторная работа 5. Docker Volumes and Networks

### Цель работы

Изучить работу Docker с томами и сетями:

- bind mount;
- named volume;
- совместное использование томов;
- стандартные и пользовательские сети;
- режим `host`.

### Ход выполнения

#### 1. Подготовка среды

На машине был установлен Docker и проверена его работоспособность.  
После этого был подготовлен рабочий каталог лабораторной и создан сценарий автоматизированного выполнения.

Подготовленные файлы:

- `lab5/work/run.sh`
- `lab5/work/index.html`

#### 2. Проверка bind mount

Был создан каталог с HTML-файлом, который затем подключался в контейнер `nginx` через bind mount.  
Контейнер был запущен с публикацией порта на хостовую машину.

Результат проверки:

- веб-страница успешно открывалась с хоста;
- содержимое HTML-файла, расположенного на хосте, корректно отображалось через `nginx`.

#### 3. Проверка named volume

Были созданы именованные тома Docker.  
Один из томов использовался как источник данных для `nginx`.

Результат проверки:

- данные из named volume успешно читались контейнером;
- при обращении к опубликованному порту `nginx` возвращал ожидаемый HTML-ответ.

#### 4. Совместное использование volume между контейнерами

Были запущены два контейнера:

- первый контейнер записывал файл в общий том;
- второй контейнер читал тот же файл из этого же тома.

Результат проверки:

- файл, созданный одним контейнером, был доступен для чтения из другого контейнера;
- это подтвердило корректную работу общего тома.

#### 5. Проверка сетей Docker

Сначала были проверены возможности стандартной сети `bridge`.  
После этого была создана пользовательская сеть `bridge` с собственной подсетью.

Результат проверки:

- в стандартной конфигурации имя контейнера не резолвилось автоматически;
- в пользовательской сети контейнеры успешно взаимодействовали друг с другом по имени;
- сетевое подключение и обмен данными между контейнерами работали корректно.

#### 6. Проверка режима host network

Контейнер был запущен в режиме `host`, внутри него был поднят простой HTTP-сервис.

Результат проверки:

- сервис был доступен напрямую через сетевой стек хоста;
- публикация порта через `-p` в этом режиме не требовалась.

### Итоги по лабораторной работе 5

В ходе лабораторной были успешно проверены основные механизмы хранения и сетевого взаимодействия Docker:

- bind mount;

- общий доступ к данным между контейнерами;
- пользовательские docker-сети;

## Лабораторная работа 6. Docker Compose

### Цель работы

Изучить сборку и запуск многоконтейнерного приложения при помощи Docker Compose, включая:

- frontend;
- backend;
- базу данных PostgreSQL;
- сохранение данных в volume;
- разделение контейнеров по сетям.

### Ход выполнения

#### 1. Подготовка проекта

Был подготовлен compose-проект, включающий три сервиса:

- `front` — веб-сервер `nginx`;
- `back` — backend-приложение на Go;
- `db` — база данных PostgreSQL.

Подготовленные файлы:

- `lab6/work/compose.yaml`
- `lab6/work/index.html`
- `lab6/work/default.conf`
- `lab6/work/.env`
- `lab6/work/users.sql`
- `lab6/work/main.go`
- `lab6/work/Dockerfile`
- `lab6/work/run.sh`

#### 2. Запуск compose-окружения

Проект был собран и запущен через Docker Compose с автоматической сборкой backend-образа.

Дополнительно были выполнены стандартные команды проверки:

- просмотр списка compose-проектов;
- проверка состояния контейнеров;
- просмотр процессов внутри контейнеров.

Результат проверки:

- все три сервиса успешно стартовали;
- compose-окружение было развёрнуто корректно.

#### 3. Проверка frontend

Frontend был опубликован на хостовой машине и проверен через HTTP-запрос.

Результат проверки:

- страница успешно открывалась в браузероподобной проверке через CLI;
- отображалась пользовательская HTML-страница;
- frontend корректно обращался к backend по маршруту `/back`.

#### 4. Проверка backend и базы данных

После запуска backend возвращал JSON-список пользователей, прочитанный из PostgreSQL.  
Затем в базу данных была добавлена новая запись через `psql`.

Результат проверки:

- до вставки отображались начальные записи;
- после добавления нового пользователя API начал возвращать обновлённый список;
- это подтвердило корректную связку `front -> back -> db`.

#### 5. Проверка сохранности данных

Для базы данных был подключён именованный том `db-data`.  
После этого контейнеры были удалены и подняты повторно.

Результат проверки:

- данные в PostgreSQL сохранились после пересоздания контейнеров;
- ранее добавленная запись осталась доступной;
- это подтвердило корректную работу постоянного тома.

#### 6. Проверка сетевой изоляции

Контейнеры были разделены на две сети:

- `front`;
- `db`.

Сервис `back` был подключён к обеим сетям, а `front` и `db` остались изолированными друг от друга.

Результат проверки:

- frontend не имел прямого доступа к контейнеру базы данных;
- попытка обращения из `front` к `db` завершалась ошибкой разрешения имени;
- требуемая изоляция сетей была достигнута.

### Итоги по лабораторной работе 6

В ходе лабораторной работы был успешно развёрнут и проверен многоконтейнерный проект на Docker Compose.  
Подтверждена корректная работа:

- frontend-сервиса;
- backend-приложения на Go;
- PostgreSQL;
- постоянного тома для БД;
- сетевого разделения сервисов.
