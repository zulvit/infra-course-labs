## Цель

Разобраться с томами Docker (bind mount и именованные тома), совместным использованием тома несколькими контейнерами и поведением сети (bridge по умолчанию, пользовательская bridge, при необходимости host).

## Ход работы

Подготовлены скрипты и статика в `work` (см. `session_lab5.typescript` в `artifacts/logs` — там полная последовательность команд).

Кратко по сценарию:

- nginx с bind mount на каталог с `index.html`, публикация на порту **18088** (8888 на хосте был занят);
- именованные тома `lab5-empty`, `lab5-new`, `lab5-html`, проверка через `volume inspect` и снова nginx на **18089**;
- запись файла из одного контейнера и чтение из второго в общий том;
- сеть: на дефолтном bridge имя контейнера не резолвится; на своей `lab5-br` с подсетью `10.0.0.0/24` ping по имени проходит;
- опционально контейнер с `network host` и `python -m http.server` на порту хоста.

## Замечания

Разделы про NFS и multi-host ipvlan из полного методического текста на одной машине не воспроизводил — для этого нужна отдельная топология из нескольких хостов.

## Вывод

Bind mount и named volume работают, том реально шарится между контейнерами. DNS имён на пользовательской bridge есть, на дефолтной — нет. Host network даёт сервис на порту машины без `-p`.
