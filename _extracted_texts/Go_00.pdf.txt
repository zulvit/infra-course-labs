Почему возник Go
• Сентябрь 2007 года. Google. Роб Пайк ждёт, пока 
скомпилируется C++ проект - 45 минут! В этот момент он думает 
не о синтаксисе нового языка, а о том, что вся модель разработки 
масштабного ПО уже сломана!
• Три человека - Rob Pike, Ken Thompson и Robert Griesemer
начинают проектировать язык. 
• Ни одна идея не попадала в спецификацию, пока все трое не 
соглашались, что она верна
• Griesemer позже заметил, что язык получился лучше, чем тот, 
который каждый из них спроектировал бы в одиночку

Проблема, которую решает Go
• Go - не ответ на вопрос «Как сделать язык мощнее?». 
• Go - ответ на вопрос «Как инженерить программное 
обеспечение при масштабе Google?»
• Конкретные боли:
• Время сборки - десятки миллионов строк C++, часы компиляции даже на 
распределённых кластерах
• Управление зависимостями - #include в C/C++ механизм включения 
исходного кода, а не модульности. Каскадные пересборки неизбежны
• Когнитивная нагрузка - C++ стал языком, в котором нужно знать язык, 
чтобы читать язык. Спецификация C++20 — ~1800 страниц. 
Спецификация Go — ~100 страниц
• Команды, не гении - код в Google пишут тысячи инженеров. Язык 
должен быть понятен среднему разработчику в команде, а не только 
автору

value semantics vs pointer semantics
• Одно из самых глубоких решений, которое Go-разработчик 
принимает для каждого типа:
• Value Semantics (семантика значений):
• Данные копируются при передаче
• Каждая функция работает со своей копией
• Безопасно для конкурентности по умолчанию
• Данные остаются на стеке → меньше давления на GC
• Pointer Semantics (семантика указателей):
• Данные разделяются через указатель
• Одна копия, множество читателей/писателей
• Эффективно для больших структур
• Требует координации при конкурентном доступе

// Горутина — не поток ОС. Это задача, управляемая Go runtime.
// Стоимость создания: ~2-8 KB (vs ~1-8 MB для OS thread)
go processOrder(order)
// Каналы — примитив синхронизации и коммуникации
ch := make(chan Order, 100) // буферизированный канал
// Producer
go func() {
for _, o := range orders {
ch <- o // отправка — может блокировать
}
close(ch)
}()
// Consumer
for order := range ch {
process(order) // чтение — блокирует до получения данных
}

Планировщик Go
• Go runtime содержит свой планировщик (M:N scheduling):
• G (Goroutine) → логическая задача
• M (Machine) → поток ОС
• P (Processor) → контекст выполнения, привязанный к

• Тысячи горутин мультиплексируются на несколько 
потоков ОС. 
• Переключение контекста горутин - наносекунды (vs
микросекунды для потоков ОС), потому что оно 
происходит в userspace, а не через kernel.

Go Proverbs- принципы языка от создателя
• Конкурентность и коммуникация:
• «Don't communicate by sharing memory, share memory
by communicating»
• В Java/C++ вы создаёте shared memory и защищаете её 
мьютексами. В Go вы передаёте владение данными через 
каналы. Данные в один момент времени принадлежат 
одной горутине - гонки данных невозможны по дизайну, а 
не по дисциплине кода

Go Proverbs- принципы языка от создателя
• «Concurrency is not parallelism»
• Конкурентность - это структура программы (как вы 
организуете независимые задачи). Параллелизм - это 
выполнение (одновременная работа на нескольких ядрах). 
Можно иметь конкурентную программу на одном ядре. Go 
даёт вам инструменты структурирования - а планировщик 
сам решает, параллелить ли
• «Channels orchestrate; mutexes serialize»
• Каналы - для координации потока работы между 
горутинами. Мьютексы - для защиты доступа к 
разделяемому состоянию. Это не взаимозаменяемые 
инструменты, у каждого своя роль

Go Proverbs- принципы языка от создателя
• «The bigger the interface, the weaker the abstraction»
• В Java принято создавать интерфейсы с десятками методов. В 
Go культура - маленькие интерфейсы. io.Reader - один метод. 
io.Writer - один метод. fmt.Stringer - один метод. Чем меньше 
интерфейс, тем больше типов его удовлетворяют, тем больше 
компонентов можно собрать вместе.

Go Proverbs- принципы языка от создателя
• «Make the zero value useful.»
• Когда вы объявляете переменную без инициализации - она получает 
нулевое значение. Go требует, чтобы это нулевое значение было 
осмысленным
// sync.Mutex — не нужен конструктор, нулевое значение = разблокированный мьютекс
var mu sync.Mutex
mu.Lock() // работает без инициализации
// bytes.Buffer — нулевое значение = пустой буфер, готовый к записи
var buf bytes.Buffer
buf.WriteString("hello") // работает без инициализации
// Ваши типы тоже должны следовать этому принципу:
type Server struct {
Addr string // "" = будет слушать на :8080 (дефолт)
Timeout time.Duration // 0 = без таймаута или разумный дефолт
}

Go Proverbs- принципы языка от создателя
• «Gofmt's style is no one's favorite, yet gofmt is everyone's favorite.»
• Никто не считает стиль gofmt идеальным. Но все счастливы, что споры о 
стиле закончились навсегда. Один стиль для всех Go-проектов в мире. Это

• «A little copying is better than a little dependency.»
• Это контринтуитивно для разработчиков из экосистемы npm/Maven. В Go 
лучше скопировать 20 строк утилитарного кода, чем тащить зависимость, 
которая потянет транзитивные зависимости, может сломаться при 
обновлении, усложнит аудит безопасности.

Go Proverbs- принципы языка от создателя
• «Clear is better than clever.»
• Есть языки, которые прославляют хитрость (Perl, Haskell, порой Rust). Go 
прославляет ясность. Если код требует комментария «тут мы делаем 
хитрый трюк» — перепишите его так, чтобы трюк не был нужен.
• «Don't just check errors, handle them gracefully.»
• Начинающие Go-разработчики пишут if err != nil { return err } механически. 
Это проверка, но не обработка. Обработка — это решение: повторить? 
вернуть дефолт? залогировать и продолжить? обернуть контекстом?

Нет исключений - и это осознанный выбор
• Исключения создают скрытый поток управления. 
• Любая строка может бросить исключение. Вы не видите этого 
в сигнатуре функции (checked exceptions в Java - попытка 
решить это, которая создала свои проблемы)
// Java - где здесь может быть исключение? Где угодно.
User user = userService.findById(id); // может бросить
String name = user.getName(); // может бросить NPE
logger.info("Found: " + name); // может бросить
// Go: поток управления всегда виден
user, err := userService.FindByID(id)
if err != nil {
return fmt.Errorf("finding user %s: %w", id, err)
}
// Если мы дошли сюда — user гарантированно валиден
