Файл init/01_schema.sql
Скрипты из папки init/ выполняются автоматически при первом старте контейнера и в алфавитном порядке. Нумерация в именах файлов гарантирует правильный порядок если их будет несколько.
Сначала создаём схему. Схема в PostgreSQL это пространство имён внутри базы данных. Для нашего примера со схемой для интернет-магазина все объекты кладём в схему shop, а не в public (это позволяет не смешивать объекты разных приложений и удобно управлять правами). Итак, в файле:
CREATE SCHEMA IF NOT EXISTS shop;
SET search_path TO shop, public;
Оборачиваем весь скрипт в транзакцию. Если что-то упадёт в рамках лабы будет неприятно, поэтому так откатится всё целиком, а не половина таблиц будет висеть мертвым грузом.

Товары:
CREATE TABLE IF NOT EXISTS shop.products (
    product_id      BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    category_id     BIGINT NOT NULL,
    manufacturer_id BIGINT NOT NULL,
    product_name    VARCHAR(255) NOT NULL,

    CONSTRAINT fk_product_category

Кому интересно - ON DELETE RESTRICT не даст удалить категорию, пока в ней есть товары. ON DELETE CASCADE удалил бы товары вместе с категорией и так не носят, опасно в рабочей системе. Лабораторная работа №2. Логическое резервирование
В данной работе в уже развёрнутую базу из работы №1 загружаются данные из CSV-файлов.
Что такое COPY и зачем он нужен?
Обычный INSERT передаёт каждую строку отдельным сообщением по протоколу PostgreSQL. COPY читает файл целиком и передаёт данные одним потоком - без overhead на разбор отдельных запросов и без промежуточных commit на каждую строку. На больших объёмах разница в скорости реально ощутима 
COPY выполняется на стороне сервера и читает файл с диска сервера. Именно поэтому CSV-файлы нужно передать внутрь контейнера через volume.

Лабораторная работа №3. Логическое резервирование
В данной работе производится логическое резервное копирование базы данных из работы №2, изучаются форматы дампа и восстановление в новую БД.
Как устроен pg_dump изнутри
pg_dump не блокирует таблицы для записи. Он открывает транзакцию с уровнем изоляции REPEATABLE READ - это даёт консистентный снимок данных на момент старта дампа. Пока дамп идёт, приложение продолжает писать в базу, но в дамп попадёт состояние на момент его начала.
На таблицы берётся лёгкий лок ACCESS SHARE - он только запрещает DROP TABLE во время дампа, но не мешает INSERT/UPDATE/SELECT.

Формат
Флаг
Особенность

Создание резервной копии
Custom-формат
docker exec shop_postgres pg_dump `
  --username=shop_admin `
  --dbname=shop_db `
  --schema=shop `
  --format=custom `
  --compress=9 `
  --file=/var/lib/postgresql/backups/shop_backup.dump `
  --verbose
Флаги:
--schema=shop - дампим только нашу схему, не системные объекты pg_catalog и information_schema
--compress=9 - максимальное сжатие, файл меньше, но чуть дольше
--verbose - выводит список объектов по мере дампа
После выполнения файл появится в папке backups на вашем хосте.
Plain SQL (бекапимся с sql)
docker exec shop_postgres pg_dump `
  --username=shop_admin `
  --dbname=shop_db `
  --schema=shop `
  --file=/var/lib/postgresql/backups/shop_backup.sql
Откройте backups\shop_backup.sql и изучите структуру. Обратите внимание на порядок секций:
SET - настройки сессии восстановления
CREATE TABLE - сначала структура без ограничений
COPY - данные
ALTER TABLE ... ADD CONSTRAINT — ограничения добавляются в конце
CREATE INDEX - индексы тоже в конце
Ограничения и индексы идут после данных намеренно: строить индекс по уже загруженным данным быстрее чем обновлять его при каждой вставке строки. Та же логика по которой мы вызывали ANALYZE после COPY.
Просмотр содержимого дампа без восстановления
docker exec shop_postgres pg_restore `
  --list /var/lib/postgresql/backups/shop_backup.dump
Вы увидите список всех объектов в дампе - таблицы, ограничения, индексы, последовательности. Это удобно чтобы убедиться что дамп полный перед восстановлением, или чтобы восстановить только один конкретный объект.
Восстановление в новую БД
Создайте новую базу данных:
docker exec shop_postgres createdb `
  --username=shop_admin `
  shop_db_restored
Восстановите из custom-дампа:
docker exec shop_postgres pg_restore `
  --username=shop_admin `
  --dbname=shop_db_restored `
  --no-owner `
  --verbose `
  /var/lib/postgresql/backups/shop_backup.dump
Кому интересно --no-owner - не восстанавливать владельцев объектов. Если дамп делал один пользователь, а восстанавливает другой - без этого флага pg_restore попытается выдать ALTER TABLE ... OWNER TO &lt;исходный_пользователь&gt; и упадёт с ошибкой прав.
Проверьте что данные восстановились:
docker exec -it shop_postgres psql -U shop_admin -d shop_db_restored
SET search_path TO shop;

-- Сравниваем количество строк с оригиналом

UNION ALL
SELECT 'categories', count(*) FROM categories
UNION ALL
SELECT 'stores', count(*) FROM stores
UNION ALL
SELECT 'customers', count(*) FROM customers;

-- Проверяем что ограничения восстановились

Лабораторная работа №4. Физическая репликация
В данной работе поднимается streaming replication. Мастер принимает запись, реплика получает WAL-поток и применяет изменения в реальном времени.
PostgreSQL записывает каждое изменение в WAL (Write-Ahead Log) - журнал операций в $PGDATA/pg_wal/. Реплика подключается к мастеру как обычный клиент, получает WAL-поток и применяет его к своим данным. Это и есть streaming replication.
Слот репликации гарантирует что мастер не удалит WAL-сегменты, пока реплика их не получила. Если реплика надолго отключится - WAL на мастере будет накапливаться и может заполнить диск. В работе мы это проверим.

На slave .env передаёт переменные окружения внутрь контейнера. Сама БД на slave не создаётся через initdb — данные приходят через pg_basebackup.

Конфигурация мастера
config/master/pg_hba.conf:

# Локальный суперпользователь
local all postgres peer
# Локальный доступ для shop_admin (Unix socket — init-скрипты, healthcheck)
local all shop_admin md5
# Приложения — только по паролю, только localhost
host shop_db shop_admin 127.0.0.1/32 scram-sha-256
host shop_db shop_admin 172.28.0.0/24 scram-sha-256
# Репликация — только пользователь replicator, только из подсети контейнеров
host replication replicator 127.0.0.1/32 scram-sha-256
host replication replicator 172.28.0.0/24 scram-sha-256

config/master/postgresql.conf:

# WAL
wal_level = replica
# replica достаточно для физической репликации.
# logical нужен только для логической репликации (Debezium, pglogical).

listen_addresses = '*'
# Принимать подключения от других контейнеров в Docker-сети.
# Доступ ограничивается через pg_hba.conf.

# Репликация
max_wal_senders = 5
max_replication_slots = 5
wal_keep_size = 512MB
# Хранить 512MB WAL дополнительно — страховка на случай кратковременного
# отключения реплики без риска заполнить диск.

# Реплика может отвечать на SELECT
hot_standby = on
hot_standby_feedback = on
# Реплика сообщает мастеру о длинных запросах — мастер не удалит строки,
# которые ещё читает реплика.

# Защита от заполнения диска слотами
max_slot_wal_keep_size = 2GB
# Если реплика не забирает WAL и накопилось &gt;2GB — слот инвалидируется.
# Реплике потребуется повторная синхронизация, но диск не заполнится.

Init-скрипты
init/03_setup_replication.sh — создаёт пользователя для репликации и слот (выполняется автоматически при первой инициализации мастера):
#!/bin/bash
set -e

psql -v ON_ERROR_STOP=1 \
    --username "$POSTGRES_USER" \
    --dbname "$POSTGRES_DB" &lt;&lt;-EOSQL
    DO \$\$
    BEGIN
        IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'replicator') THEN
            CREATE USER replicator
                WITH REPLICATION
                ENCRYPTED PASSWORD 'rut_replicator_password';
        END IF;
    END \$\$;

    SELECT pg_create_physical_replication_slot('replication_slot_slave1', true)
    WHERE NOT EXISTS (
        SELECT FROM pg_replication_slots
        WHERE slot_name = 'replication_slot_slave1'
    );
EOSQL
Скрипт идемпотентный - безопасно запускать повторно. Проверяет существование роли и слота перед созданием.
Entrypoint реплики
config/slave/entrypoint.sh - кастомный entrypoint для контейнера реплики. При первом запуске автоматически выполняет pg_basebackup с мастера:
#!/bin/bash
set -e

# Если data-директория пуста — нужен pg_basebackup с мастера
if [ -z "$(ls -A /var/lib/postgresql/data)" ]; then
    echo "&gt;&gt;&gt; Data directory is empty. Running pg_basebackup from master..."

    PGPASSWORD=rut_replicator_password pg_basebackup \
        --host=postgres_master \
        --port=5432 \
        --username=replicator \
        --pgdata=/var/lib/postgresql/data \
        --slot=replication_slot_slave1 \
        --wal-method=stream \
        --checkpoint=fast \
        --write-recovery-conf \
        --progress

    echo "&gt;&gt;&gt; pg_basebackup complete."

    chmod 0700 /var/lib/postgresql/data
fi

# Запускаем postgres с нашим конфигом
exec postgres \
    -c config_file=/etc/postgresql/postgresql.conf \
    -c hba_file=/etc/postgresql/pg_hba.conf
pg_basebackup запускается со стороны реплики и подключается к мастеру по имени postgres_master (через Docker-сеть). Благодаря этому --write-recovery-conf записывает правильный host=postgres_master в postgresql.auto.conf.
--wal-method=stream - WAL передаётся параллельно с данными. Бэкап сразу готов к использованию, не нужно ждать конца копирования.
--write-recovery-conf - автоматически создаёт файл standby.signal и записывает primary_conninfo в postgresql.auto.conf.

Ключевые моменты:
Мастер использует стандартный entrypoint PostgreSQL. При первой инициализации автоматически выполняются init-скрипты из ./init/, включая создание пользователя replicator и слота репликации.
Реплика использует кастомный entrypoint.sh. При пустой data-директории выполняет pg_basebackup с мастера, затем запускает postgres.
depends_on: condition: service_healthy гарантирует что реплика стартует только после полной инициализации мастера.
user: postgres нужен для реплики, чтобы pg_basebackup мог записать данные в /var/lib/postgresql/data.
Запуск
Запустите оба контейнера одной командой:
docker-compose up -d
Docker Compose сначала дождётся пока мастер станет healthy (init-скрипты выполнятся, БД будет готова), затем запустит реплику. Реплика автоматически выполнит pg_basebackup и начнёт получать WAL-поток.
Проверка репликации
Проверьте состояние на мастере:
docker exec -it postgres_master psql -U shop_admin -d shop_db
SELECT application_name, state, pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn)) AS replication_lag, replay_lag, sync_state FROM pg_stat_replication;
Вы должны увидеть строку с application_name = slave1 и state = streaming.
Проверьте на реплике:
docker exec -it postgres_slave psql -U shop_admin -d shop_db
SELECT
    pg_is_in_recovery()              AS is_replica,
    pg_last_wal_replay_lsn()         AS replayed_lsn,
    NOW() - pg_last_xact_replay_timestamp() AS lag;
Проверка синхронизации
На мастере вставьте строку:
SET search_path TO shop;
INSERT INTO manufacturers(manufacturer_name) VALUES ('Test Vendor');
SELECT * FROM manufacturers ORDER BY manufacturer_id DESC LIMIT 3;
На реплике проверьте что она появилась:
SET search_path TO shop;
SELECT * FROM manufacturers ORDER BY manufacturer_id DESC LIMIT 3;
Попробуйте на реплике сделать INSERT:
INSERT INTO manufacturers(manufacturer_name) VALUES ('Should fail');
Вы получите ошибку cannot execute INSERT in a read-only transaction - реплика только для чтения.
Остановка мастера и наблюдение
Остановите контейнер мастера:
docker-compose stop postgres_master
Откройте логи реплики:
docker-compose logs -f postgres_slave
Вы увидите попытки переподключения к мастеру. Реплика продолжает отвечать на SELECT - это и есть hot_standby = on.
Запустите мастер обратно:
docker-compose start postgres_master
Репликация восстановится автоматически.
