# Промпт для Codex: финальный отчёт + DOCX через otchet-compose

Используй **после** того, как выполнены (или честно задокументированы) лабораторные `lab1`–`lab8`. Задача — по **каждой** лабе отдельно оформить материалы из `labN/task.md`, `labN/report.md` и фрагментов `labN/artifacts/logs/` и сгенерировать **восемь отдельных** файлов Word (DOCX) утилитой [otchet-compose](https://github.com/sorivma/otchet-compose) — **по одному DOCX на лабу** (`lab1` … `lab8`). Сводный «один отчёт на весь курс» **не** делать, если пользователь явно не попросит обратное.

---

## 0. Утилита otchet-compose (локально, не в git)

В корне проекта (рядом с папками `lab1`, `lab2`, …) должна лежать клонированная копия:

```bash
git clone https://github.com/sorivma/otchet-compose.git otchet-compose
```

Каталог `otchet-compose/` **не коммитится** (уже в `.gitignore`).

Установка и запуск (из корня проекта):

```bash
cd otchet-compose
pip install -e .
cd ..
```

Проверка:

```bash
otchet-compose gen -f path/to/config.yml
```

Если конфиг лежит в текущей директории как `otchet-compose.yml`, достаточно:

```bash
otchet-compose gen
```

Итоговый файл создаётся по пути `document.output` из YAML; каталоги создаются автоматически.

Официальные подсказки по формату YAML для нейросети лежат в репозитории утилиты:

- `otchet-compose/doc/prompt.txt` — правила для LLM;
- `otchet-compose/doc/otchet-compose.llm.ref.yml` — развёрнутый эталон с комментариями;
- `otchet-compose/examples/otchet-compose.yml` — рабочий пример.

## 1. Что прочитать из проекта (источники правды)

Для каждой лабораторной `labN`:

1. **`labN/task.md`** — тема, ожидаемый результат, смысл работы (что *должно* было быть по курсу).
2. **`labN/report.md`** — что сделано, блокеры, итог.
3. **`labN/artifacts/logs/`** — **не копировать целиком**, но **обязательно встроить в отчёт сами фрагменты**, а не только пути к файлам.

**Критично для читателя:** отчёт сдают «глазами». Преподаватель **не обязан** открывать `lab3/artifacts/logs/step05.out`. Поэтому выбранные фрагменты логов и вывода команд должны **появиться в теле DOCX** — в абзацах или в ячейках таблицы (колонка «Вывод», «Фрагмент лога», «Результат выполнения» и т.п.). Путь к файлу можно указать **дополнительно** в скобках или сноской для трассируемости, но **недостаточно** писать одну фразу вроде «лог см. в `lab1/artifacts/logs/step06.out`» без текста лога.

Выбирай для вставки:
   - команды, которые реально отражают ход работы;
   - **короткие, но содержательные** фрагменты stdout/stderr (обычно от нескольких строк до ~15–25 строк на эпизод), подтверждающие успех или характер ошибки;
   - итоговые проверки (`curl`, `docker ps`, `vagrant status`, и т.д.) — **с реальным выводом**, не только «команда выполнена».

Игнорируй:

- повторяющиеся попытки одной и той же команды;
- длинные логи сборок, если в отчёте достаточно факта «образ собран / сервис запущен»;
- всё, что не относится к формулировкам из `task.md`.

При наличии скриншотов: `labN/assets/screenshots/*.png` — можно сослаться в блоках `figure` (путь относительно YAML, см. ниже).

## 2. Структура каждого из восьми отчётов (содержательно)

Для **каждого** `labN` получается **свой** DOCX: только эта лабораторная, без пересказа остальных лаб.

Внутри документа (как и требует otchet-compose) сохраняй три структурных раздела уровня 1:

1. **«Цель работы»** — введение и цели **именно этой** работы по смыслу `labN/task.md` (тема, инструменты, ожидаемый результат). Кратко упомяни среду выполнения, если из `report.md` следует, что она влияла на результат (ОС, WSL, блокировка виртуализации и т.д.).
2. **«Ход работы»** — только шаги **этой** лабы:
   - краткая последовательность (3–7 пунктов, не пересказ всех логов подряд);
   - **таблица** «Этап / Команда / Результат (краткий вывод + фрагмент вывода)» — в колонке результата **встроенный текст** из логов, не только «успешно»;
   - при необходимости **абзацы** с фрагментами лога (ошибки, `curl`, вывод `docker`/`vagrant` и т.д.);
   - при блокере — **цитата** из stderr/лога + пояснение; путь к полному логу в скобках опционально.
   - подзаголовки уровня 2 (`level: 2`) — по смыслу этапов **этой** лабы (например «Установка окружения», «Запуск контейнеров», «Проверка»), а не «Лабораторная 2», «Лабораторная 3».
3. **«Заключение»** — итоги **только** по этой работе: что сделано, что подтверждено, что не вышло в среде (если было).

**Баланс:** не вставляй многостраничные дампы. Вставляй **достаточно строк**, чтобы читатель без открытия файлов понял, *что именно* произошло.

## 3. Формат YAML для otchet-compose (строго)

Соблюдай схему из `otchet-compose/doc/prompt.txt` и пример `examples/otchet-compose.yml`:

- `version: 1`
- **`document.output`** для лабы `labN`: `./build/labN-report.docx` (каталог `labN/build/` создаётся рядом с конфигом).
- **Запуск генерации:** заходи в папку лабы и вызывай утилиту оттуда, чтобы пути в YAML совпадали с рабочей директорией:

  ```bash
  cd labN
  otchet-compose gen -f otchet-compose.yml
  cd ..
  ```

  Либо одной строкой из корня (PowerShell): `Set-Location lab1; otchet-compose gen -f otchet-compose.yml; Set-Location ..`
- `document.toc: true`
- Титульный лист: `document.title_page`, шаблон `rut` или `mock`. В `work_number` укажи номер лабы (**«№ 1»** … **«№ 8»**), в `work_title` — краткое название из темы `task.md` этой лабы. Остальные `params` (ФИО, группа, преподаватель, дисциплина, год) — единые для всех восьми файлов или плейсхолдеры `TODO_...`.
- При использовании `title_page` **не** ставь `reserve_title_page: true`.

Обязательные структурные разделы (`level: 1`, `structural: true`) **в каждом** из восьми файлов:

- «Цель работы»
- «Ход работы»
- «Заключение»

Внутри «Хода работы» — подзаголовки `level: 2`, `structural: false` только по этапам **данной** лабораторной.

Типы блоков **только** поддерживаемые: `heading`, `paragraph`, `figure`, `list`, `table`.

- Текст абзацев — в `paragraph.text` в виде folded scalar `>`.
- Фрагменты логов вставляй **внутрь** `paragraph.text` или в ячейки `table.rows` (как обычный текст). Если строк много, разбей на **несколько** блоков `paragraph` подряд или на несколько строк одной ячейки (как единый текст с пробелами/точками с запятой между строками — главное, чтобы в Word это читалось). **Не** оставляй вместо содержимого только путь к `.out`/`.err`.
- Списки — `type: list`, `style: bullet` или `numeric`.
- Подпись рисунка **без** номера («Рисунок 1» добавит генератор). В тексте перед рисунком можно отослаться к нему словами «на рисунке ниже».
- `figure.path` — путь **относительно YAML-файла**. Если конфиг — `labN/otchet-compose.yml`, удобно `./assets/screenshots/...` (внутри той же папки `labN`). Если файла нет — блок `figure` без `path` (будет placeholder).

## 4. Выходные артефакты, которые ты создаёшь

1. **Восемь** конфигураций: `lab1/otchet-compose.yml`, `lab2/otchet-compose.yml`, …, `lab8/otchet-compose.yml`.
2. **Восемь** DOCX: согласно `document.output` в каждом YAML (например `labN/build/labN-report.docx`). После правки титульных данных пользователем — повторный `gen` только для затронутых лаб.
3. Запуск генерации **всех** отчётов (bash, из корня репозитория):

   ```bash
   for n in 1 2 3 4 5 6 7 8; do (cd "lab$n" && otchet-compose gen -f otchet-compose.yml); done
   ```

   PowerShell:

   ```powershell
   1..8 | ForEach-Object { Set-Location "lab$_"; otchet-compose gen -f otchet-compose.yml; Set-Location .. }
   ```

4. Краткий **`REPORTS_DOCX.md`** в корне: таблица «лаба → YAML → DOCX», как запускать `gen`, какие `TODO` в титулах заменить.

## 5. Ограничения и аккуратность

- Не выдумывай выполнение шагов, которых нет в `report.md` / логах.
- Любая ссылка на лог должна сопровождаться **встроенной цитатой** из него (см. раздел 1); путь к файлу — опционально, не вместо цитаты.
- Не включай секреты (токены, пароли из `.env`).
- Сохраняй нейтральный академический стиль, без разговорных оборотов.
- Один общий DOCX на весь курс делай **только** если пользователь явно попросит; по умолчанию — **всегда восемь** отдельных файлов.

## Краткая шпаргалка команд для пользователя (после твоей работы)

```bash
# из корня проекта (где lab1, lab2, … и папка otchet-compose)
pip install -e ./otchet-compose
for n in 1 2 3 4 5 6 7 8; do (cd "lab$n" && otchet-compose gen -f otchet-compose.yml); done
```

Открыть каждый `labN/build/labN-report.docx` в Word / LibreOffice и при необходимости поправить титульный лист.
